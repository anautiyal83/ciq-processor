Part 1 — Project Overview
①What Is It?
A self-contained Java 8+ project that reads a Nokia CIQ Excel workbook directly into memory, validates every cell against configurable YAML rules, and produces structured validation reports. No intermediate files are written during processing — the entire read → validate → report pipeline runs in JVM heap.
Java 8+Apache POISnakeYAML
②What Does It Validate?
Validates Nokia CIQ (Configuration Information Quality) Excel workbooks used to configure network elements. Checks that every row in every configuration table meets the rules defined in the YAML rules file — required fields, allowed values, numeric ranges, string patterns, length limits, and cross-sheet foreign-key references.

Produces up to four types of output depending on the arguments provided:

Reads the CIQ Excel workbook using Apache POI and stores all data in JVM heap. Three-phase read: Node_ID sheet → NIAM mapping; Index sheet → CiqIndex (node/CRGroup/tables); each table sheet → CiqSheet with rows and column names. ①RequiredValidator
YAML keyrequired: true
Acts as a gatekeeper. If a column is marked required and the cell is blank, an error is recorded and all subsequent validators for that cell are skipped. This prevents misleading downstream errors on blank cells.
②AllowedValuesValidator
YAML keyallowedValues: [A, B, C]
Validates that the cell value is one of the explicitly listed allowed values. Case-sensitive. Suitable for action codes (CREATE/MODIFY/DELETE), enumeration fields, and fixed-set values.
⑥PatternValidator
YAML keypattern: "^[A-Z0-9_]+$"
Applies a Java regex pattern to the cell value. Used for format validation: IP addresses, hostnames, service names, identifiers. Non-matching values are reported with the optional patternMessage from the YAML rule.

Part 8 — Excel Parsing Details
1Node_ID Sheet
PurposeBuilds NIAM (node → login ID) mapping
Columns readNode, NIAM
Locates the header row by scanning the first 10 rows for both "Node" and "NIAM" column names. Builds a map of node name → NIAM login. Stored in CiqIndex.niamMapping and available to downstream consumers.
2Index Sheet
PurposeBuilds node / CRGroup / table list
Columns readNode, CRGroup, Tables
Reads only the FIRST "Tables" column (the user-selection column). A second "Tables" column present after blank columns is a dropdown catalog/reference and is intentionally skipped. Groups rows by Node+CRGroup key into NodeEntry objects, each holding a list of table names.
3Data Sheets
PurposeReads each table's row data
Header detectionScans first 10 rows, case-insensitive
For each table (from Index, or all non-metadata sheets when Index is absent), finds the corresponding sheet via exact match then 31-char prefix-match fallback. Header row detection is case-insensitive — tries "Node"+"Action" first, then falls back to "Node" alone for sheets without an Action column. Each data row becomes a CiqRow with its Excel row number and a column-to-value map. Column lookups are case-insensitive at validation time.
EdgeSheet Name Truncation
IssueExcel limits sheet names to 31 characters
When a table name from the Index exceeds 31 characters, Excel silently truncates the sheet name. The reader handles this with a prefix match in both directions after an exact match fails: tableName.startsWith(sheetName) || sheetName.startsWith(tableName). Tables with long names are always found.
EdgeCell Type Handling
Numeric integersFormatted without ".0" suffix
getCellString() handles all POI cell types: STRING (trimmed, null if empty), NUMERIC (whole numbers cast to long — "1.0" becomes "1"), BOOLEAN (true/false string), FORMULA (cached result evaluated), BLANK/_NONE (returns null). Null-named header columns are excluded from the column list.
Part 9 — Child-Order JSON Output (MOP Generator Input)
MOPChild-Order Segregation
TriggerValidation PASSED + --mop-json-dir argument provided
StructureOne subfolder per Node_CRGroup combination
After a PASSED validation, CiqProcessorImpl segregates the CIQ data into child-order subfolders that the MOP Generator reads directly. Each folder is named {Node}_{CRGroup} and contains a filtered index JSON plus one JSON file per configuration table:
# Directory structure produced: mop-json-dir/ Nokia-SBC-1_CRG001/ index.json # filtered CiqIndex for this node/CRGroup SBC_SUBSCRIBER_DATA.json # rows for this node/CRGroup only SBC_SUBSCRIBER_PROFILE.json SBC_ROUTE_DATA.json ... Nokia-SBC-2_CRG001/ index.json SBC_SUBSCRIBER_DATA.json ...

NOTEEnd-to-End Workflow
ciq-processor and mop-generator-utility are independent tools designed to work in sequence. ciq-processor produces the child-order JSON files that mop-generator-utility consumes via its --json-dir argument. Child-order JSON is only written when validation passes — MOP generation is never triggered on invalid CIQ data.

Dev-2 — Adding a New Cell Validator
GUIDEStep-by-Step: Add a New Validator
All cell validators follow the same pattern. Follow these four steps to add a new one.
Step 1 — Add config field to ColumnRule
// com.nokia.ciq.validator.config.ColumnRule private String myNewRule; // add getter + setter public String getMyNewRule() { return myNewRule; } public void setMyNewRule(String v) { myNewRule = v; }
Step 2 — Create the Validator class
// com.nokia.ciq.validator.validator.MyNewValidator public class MyNewValidator implements CellValidator { @Override public Optional<String> validate( String value, ColumnRule rule, CiqDataStore store) { if (rule.getMyNewRule() == null) return Optional.empty(); // return Optional.of("error message") on failure // return Optional.empty() on pass } }
Step 3 — Register in CiqValidationEngine
// com.nokia.ciq.validator.CiqValidationEngine // Add to the validators list in the constructor: validators.add(new MyNewValidator());
The engine calls each validator in list order. Place your validator after the appropriate gate (e.g., after RequiredValidator if it should skip on blank).
Step 4 — Add YAML key to the rules file
- name: "MyColumn" myNewRule: "some-value" # new key maps to ColumnRule.myNewRule via SnakeYAML
SnakeYAML automatically maps camelCase YAML keys to Java fields by name. No registration needed in the loader.

Dev-4 — Internal Data Flow
①CiqIndex
Classcom.nokia.ciq.reader.model.CiqIndex
Populated by InMemoryExcelReader from the Index sheet. Contains:
  • nodeType — e.g. "SBC"
  • activity — e.g. "FIXED_LINE_CONFIGURATION"
  • entries — List<NodeEntry> (node/CRGroup + tables)
  • niamMapping — Map<node, loginId>
  • getAllTables() — unique, ordered table name list
②CiqSheet
Classcom.nokia.ciq.reader.model.CiqSheet
One per configuration table. Contains:
  • sheetName — table name
  • columns — ordered List<String> of column names
  • rows — List<CiqRow>
Each CiqRow holds the 1-based Excel row number and a Map<String,String> of column → value. Blank cells are stored as null.
③ValidationReport
Classcom.nokia.ciq.validator.model.ValidationReport
Root result object returned by CiqProcessorImpl. Contains:
  • status — "PASSED" or "FAILED"
  • totalErrors — sum across all sheets
  • sheets — List<SheetValidationResult>
Each SheetValidationResult holds the sheet name and a list of ValidationError (row, column, value, message).
Dev-5 — Supporting a New Node Type or Activity
GUIDEHow to onboard a new Node Type / Activity combination
The processor is fully configuration-driven. No code changes are needed to add a new node type or activity — only a new YAML rules file is required.
1 — Create the YAML rules file
# File name follows the convention: # {NodeType}_{ACTIVITY}_validation-rules.yaml # Example: MSS_SUBSCRIBER_MIGRATION_validation-rules.yaml # For CIQs that have no Index sheet (e.g. MRF): validateIndexSheets: false validateNodeIds: true
Add a sheets: entry for every CIQ table to validate. When validateIndexSheets: false, the processor reads all non-metadata workbook sheets directly and validates them against the rules config — no Index sheet required.
2 — Run with the new parameters
java -jar ciq-processor-1.0.0-cli.jar \ --ciq MSS_CIQ.xlsx \ --node-type MSS \ --activity SUBSCRIBER_MIGRATION \ --rules MSS_SUBSCRIBER_MIGRATION_validation-rules.yaml \ --output reports/
3 — That's it
The processor infers the node type and activity from the CLI arguments. Report file names, child-order folder names, and all other outputs are derived automatically from --node-type and --activity. No code or configuration registration needed.

Dev-8 — Common Errors & Troubleshooting
ERRIOException: CIQ file not found
Cause--ciq path does not exist or is not readable
Thrown by validateFile() in InMemoryExcelReader before opening the workbook. Check: file exists, path uses forward slashes or escaped backslashes, file extension is .xlsx (not .xls or .xlsm).
ERRIndex sheet produces empty results
CauseNo Index sheet, or header row not found
The reader scans only the first 10 rows of the Index sheet for "Node", "CRGroup", "Tables" columns (case-insensitive). If the Index sheet is absent, all non-metadata workbook sheets are read directly. The validation engine then uses the sheets: keys from the rules config to determine which sheets to validate — set validateIndexSheets: false in the rules YAML for CIQs that have no Index sheet.
ERRData sheet loaded with 0 rows
CauseHeader row not found in data sheet
The reader first tries to locate a header row containing both "Node" and "Action" (case-insensitive), then falls back to "Node" alone. If neither is found within the first 10 rows, the sheet is loaded with 0 rows. Check the WARN log: "Header row not found in sheet 'X'". Common causes: merged cells above the header, the column labelled "NODE" instead of "Node" (handled automatically), or the header sitting below row 10.
