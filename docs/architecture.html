CIQ Processor — Architecture

Visual reference: end-to-end data flow, component layers, validation pipeline, and output modes

1   End-to-End Data Flow
Core processor
Input / data
Validation output
JSON output / downstream
YAML config files
2   Internal Component Layers
Data flows top-down through the layers. CiqProcessorImpl drives the full pipeline: it calls the reader, hands the loaded store to the validation engine, and dispatches to the appropriate report writers. JsonTemplateEvaluator is invoked only when validation passes and --json-output-dir is supplied. CiqTemplateGenerator runs only in ciq-generate mode.
3   Validation Pipeline — Step by Step
Step 3 runs three validator tiers in order: (a) per-column validators on every cell, (b) row-rule validators on every data row, (c) workbook-level validators across sheets. Errors are accumulated — all sheets are always validated regardless of earlier failures. Step 5 runs only when validation passes and both --json-output-dir and --json-output-config-file are provided.
4   JSON Template Evaluation (_each Directives)
JsonTemplateEvaluator is fully domain-agnostic — it has no hardcoded column or sheet names. The DISTINCT form is recommended: it deduplicates values and auto-scopes the named sheet's rows so that plain Sheet.Column lookups return per-value data without needing extra WHERE clauses.
5   Output Modes
Validation reports are always written to --output regardless of mode: {NODE_TYPE}_{ACTIVITY}_VALIDATION_REPORT.json / .html / .xlsx. The base name can be overridden with report_output.filename in the rules YAML.
6   CIQ INDEX Sheet — Three Detection Modes
The INDEX mode is auto-detected by InMemoryExcelReader based on which column headers are present. Mode 1 is the original per-node format. Mode 2 and Mode 3 support large multi-node rollouts where many nodes share the same configuration data.
