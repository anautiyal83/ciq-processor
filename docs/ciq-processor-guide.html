CIQ Processor — Developer Guide

Validates CIQ Excel workbooks · Produces validation reports · Generates MOP JSON output

Overview

The CIQ Processor reads a Nokia CIQ (Configuration Input Questionnaire) Excel workbook, validates every data row against YAML-defined rules, writes validation reports, and — when validation passes — segregates the data into per-scope JSON folders consumed by mop-generator-utility.

CIQ Excel (.xlsx)
▶
InMemoryExcelReader
▶
CiqValidationEngine
▶
Reports (JSON / HTML / Excel)
▶
validation PASSED + --mop-json-dir
▶
mop-json/ folders
▶

The entire pipeline runs in a single pass with no intermediate disk writes until the reports and optional MOP JSON are ready.

CLI Usage

java -jar ciq-processor-1.0.0-cli.jar [options]

Required:
  --ciq           <file>   CIQ Excel workbook (.xlsx)
  --node-type     <type>   Node type, e.g. SBC or MRF
  --activity      <name>   Activity name, e.g. FIXED_LINE_CONFIGURATION
  --rules         <file>   YAML validation-rules file
  --output        <dir>    Output directory for validation reports

Optional:
  --format        <csv>    Report formats: JSON,HTML,MSEXCEL  (default: all three)
  --mop-json-dir  <dir>    JSON output for MOP generation (only written on PASSED)
  --report-name   <name>   Base file name for reports (no extension)
                           Default: <node-type>_<activity>_validation-report

Exit code:  0 = PASSED,  1 = FAILED or error

Example — SBC NODE mode

java -jar ciq-processor-1.0.0-cli.jar \
  --ciq         SBC_FIXED_LINE_CONFIGURATION_CIQ.xlsx           \
  --node-type   SBC                                             \
  --activity    FIXED_LINE_CONFIGURATION                        \
  --rules       SBC_FIXED_LINE_CONFIGURATION_validation-rules.yaml \
  --output      target/processor-output                         \
  --format      JSON,HTML                                       \
  --mop-json-dir target/mop-json

Example — MRF CRGROUP mode

java -jar ciq-processor-1.0.0-cli.jar \

Grouping Modes

The mode is controlled by groupByColumnName in the validation-rules YAML. It determines the INDEX sheet column structure, the JSON output folder layout, and which parameter mop-generator-utility expects.

groupByColumnName INDEX sheet columns JSON output layout mop-generator-utility
CRGROUP CRGROUP GROUP | CRGROUP | NODE mop-json/<CRGROUP>/ --crgroup <id>
GROUP GROUP GROUP | NODE mop-json/<GROUP>/ --group <name>
NODE NODE Node | CRGroup | Tables mop-json/<Node>_<CRGroup>/ --host <node>

CRGROUP

  • Change window = CRGROUP
  • MOP content = GROUP
  • One folder per CRGROUP
  • GROUP may span multiple CRs

GROUP

  • Homogeneous MOP per group
  • All nodes in a GROUP get identical rows
  • One folder per GROUP
  • No CRGROUP concept

NODE

  • Heterogeneous MOP per node
  • Each node has its own rows
  • One folder per Node+CRGroup
  • Tables listed in INDEX per node

CRGROUP Mode groupByColumnName: CRGROUP

Used when nodes are organised into change requests, each change request spanning one or more GROUPs. The CRGROUP drives the change window (maintenance slot); the GROUP drives the MOP content (all nodes in a GROUP receive identical configuration rows).

Used when each node has its own individual configuration rows (heterogeneous MOP). The child-order key is <Node>_<CRGroup>. Each data sheet must have a Node column; rows are filtered per node when writing JSON.

Validation Reports

Three report formats are supported, all written to --output:

FormatFileContents
JSON <name>.json Machine-readable; full error list with sheet, row, column, message and status
HTML <name>.html Human-readable; colour-coded per-sheet error tables with summary counts
MSEXCEL <name>.xlsx Excel report; one sheet per data sheet with error rows highlighted

Default base name: <node-type>_<activity>_validation-report
Override with --report-name.

CIQ Columns as MOP Template Variables

After segregation, the per-group JSON files are consumed by mop-generator-utility. Any data sheet column becomes a ${COLUMN_NAME} variable that can be referenced in the MOP template YAML commands. Two resolution strategies apply:

Column characteristicResolved asExample
Same value across all rows in the sheet (e.g. CIRCLE, VERSION) Group-level variable — substituted once, globally ${CIRCLE} → RJ
Different value per row (e.g. INPUT_FILE, MRF_DESTINATION_PATH) Per-row expansion — command is written once per CIQ row with the row's value ${INPUT_FILE} → one command per file in the CIQ table

Built-in variables always available regardless of CIQ data: ${NODE} (node name from the GroupIndex) and ${NEID} (NE identifier from the NIAM mapping).

Column names in the CIQ data sheet must match ${COLUMN_NAME} references in the MOP template exactly (case-sensitive). Use Record.-prefixed columns (Record.INPUT_FILE) without the prefix in the template reference (${INPUT_FILE}).

For MODIFY rows, ActionKey (which field to match on) and SubAction (ADD/DEL/MOD for sub-table entries) may be required depending on the rules.

Validation Rules YAML

File naming convention: {NODE_TYPE}_{ACTIVITY}_validation-rules.yaml
Example: MRF_ANNOUNCEMENT_LOADING_validation-rules.yaml

Top-level Settings

# Controls JSON segregation mode and INDEX sheet interpretation
groupByColumnName: CRGROUP     # CRGROUP | GROUP | NODE

# Validate that every table listed in INDEX has a matching data sheet
# true for NODE mode; false for GROUP/CRGROUP mode (no Tables column)
validateIndexSheets: false

# Validate that every node in the INDEX exists in the NODE_ID sheet
validateNodeIds: true

INDEX Sheet Column Rules

indexSheet:
  columns:
    # ── CRGROUP mode ──────────────────────────────────
    GROUP:   { required: true }
    CRGROUP: { required: true }
    NODE:    { required: true }

    # ── GROUP mode ────────────────────────────────────
    # GROUP: { required: true }
    # NODE:  { required: true }

    # ── NODE mode ─────────────────────────────────────
    # Node:    { required: true }
    # CRGroup: { required: true }
    # Tables:  { required: true }

NODE_ID Sheet Column Rules

nodeIdSheet:
  columns:
    NODE:    { required: true }   # CRGROUP / GROUP mode
    NIAM_ID: { required: true }   # CRGROUP / GROUP mode

    # NODE mode uses:
    # Node: { required: true }
    # NIAM: { required: true }

Data Sheet Rules

Column reference convention — applies to all rule types:
ColumnName → column in the current sheet (default)
SheetName.ColumnName → column in the named sheet
This applies to require, forbid, when.column, compare, one_of, only_one, all_or_none, sum, and equals. 
sheets:

  MyTable:
    columns:

      Node:
        required: true
        crossRef:             # Value must exist in another sheet's column
          sheet: "_index"     # "_index" = the INDEX sheet
          column: node

      Action:
        required: true
        type: enum
        values: [CREATE, DELETE, MODIFY]

      ActionKey:
        requiredWhen:         # Required only when another column has a specific value
          column: Action
          value:  MODIFY

      SubAction:
        requiredWhen:
          column: Action
          value:  MODIFY
        type: enum
        values: [ADD, DEL, MOD]

      ID:
        required: true
        type: integer
        minValue: 1
        maxValue: 6000

      NAME:
        minLength: 1
        maxLength: 19

      Record.VERSION:
        required: true
        pattern: "^\\d+\\.\\d+\\.\\d+\\.\\d+$"
        messages:
          pattern: "VERSION must be in format X.X.X.X (e.g. 17.0.3.7)"

      Record.SOME_RANGE_FIELD:
        type: integer
        allowedRanges:        # Multiple allowed numeric bands (OR logic)
          - { min: 0,    max: 0    }
          - { min: 500,  max: 500  }
          - { min: 1024, max: 65535 }

      Tables:
        required: true
        sheetRef: true              # value must match an existing sheet name
        sheetRefIgnoreCase: false   # exact-case match (set true for case-insensitive)

    rules:

      # require a column when a same-sheet condition is met
      - require: ActionKey
        when:
          column: Action
          operator: equals          # equals | notEquals | blank | notBlank | contains
          value: MODIFY             # (not used for blank/notBlank)

      # forbid a column based on a condition
      - forbid: SubAction
        when:
          column: Action
          operator: equals
          value: CREATE

      # cross-sheet require — value must exist in referenced sheet's column
      - require: Node_Details.Node_Name

      # cross-sheet when — apply rule only when row's value exists in another sheet
      - require: NIAM_NAME
        when:
          sheet: Node_Details       # sheet to look up in
          column: Node_Name         # column whose value is looked up
          operator: exists          # exists | notExists

      # numeric/lexicographic when condition
      - require: SubAction
        when:
          column: Version
          operator: greaterThan     # greaterThan | greaterThanOrEquals | lessThan | lessThanOrEquals
          value: "17.0"

      # compound when condition (AND / OR nesting)
      - require: SubAction
        when:
          all:
            - column: Action
              operator: equals
              value: MODIFY
            - any:
                - column: SubAction
                  operator: blank
                - column: ActionKey
                  operator: notBlank

      # cross-column comparison — string operators only
      - compare: "StartPort lessThanOrEquals EndPort"

      # group presence
      - one_of: [IPv4_Address, IPv6_Address]
      - all_or_none: [SubAction, ActionKey]

      # column sum
      - sum: [Leg1, Leg2]
        equals: Total

# require unconditional — same-sheet
- require: ActionKey

# require unconditional — cross-sheet (value must exist in Node_Details.Node_Name)
- require: Node_Details.Node_Name

# require conditional — same-sheet
- require: ActionKey
  when:
    column: Action                   # ColumnName = same sheet
    operator: equals
    value: MODIFY

# require conditional — cross-sheet
- require: Node_Details.Node_Name
  when:
    column: Action
    operator: notEquals
    value: DELETE

- require: NIAM_NAME
  when:
    sheet: Node_Details
    column: Node_Name
    operator: exists

# forbid unconditional — same-sheet
- forbid: SubAction

# forbid unconditional — cross-sheet (value must NOT exist in Blacklist.ID)
- forbid: Blacklist.ID

# forbid conditional — cross-sheet
- forbid: Blacklist.ID
  when:
    column: Action
    operator: notEquals
    value: DELETE

Programmatic API

ValidationReport report = new CiqProcessorImpl().process(
    "/path/to/CIQ.xlsx",                           // ciqFilePath
    "MRF",                                         // nodeType
    "ANNOUNCEMENT_LOADING",                        // activity
    "/path/to/MRF_ANNOUNCEMENT_LOADING_validation-rules.yaml", // rulesFilePath
    "/path/to/output",                             // outputDir
    List.of(ReportFormat.JSON, ReportFormat.HTML), // formats
    "/path/to/mop-json",                           // mopJsonOutputDir (null = skip)
    null                                           // reportFileName   (null = default)
);

System.out.println(report.getStatus());        // "PASSED" or "FAILED"
System.out.println(report.getTotalErrors());   // total error count

// Output parameters (populated on PASSED)
report.getParameters().forEach((k, v) -> System.out.println(k + "=" + v));

// Per-sheet errors
for (SheetValidationResult r : report.getSheetResults()) {
    System.out.println(r.getSheetName() + ": " + r.getErrors().size() + " error(s)");
}

Adding a New Node Type / Activity

No Java code changes are needed. Only a YAML file is required.
  1. Create {NODE_TYPE}_{ACTIVITY}_validation-rules.yaml.
  2. Set groupByColumnName to match the CIQ INDEX sheet column structure (CRGROUP, GROUP, or NODE).
  3. Set validateIndexSheets and validateNodeIds for the chosen mode.
  4. Define indexSheet.columns and nodeIdSheet.columns for the workbook layout.
  5. Define sheets: with column-level rules for each data sheet.
  6. Pass the YAML path via --rules on the CLI, or as rulesFilePath in the API.

Validation rules YAML skeleton

# {NODE_TYPE}_{ACTIVITY}_validation-rules.yaml

groupByColumnName: CRGROUP   # CRGROUP | GROUP | NODE
validateIndexSheets: false
validateNodeIds: true

indexSheet:
  columns:
    GROUP:   { required: true }
    CRGROUP: { required: true }
    NODE:    { required: true }

nodeIdSheet:
  columns:
    NODE:    { required: true }
    NIAM_ID: { required: true }

sheets:

  MyFirstTable:
    columns:
      GROUP:
        required: true
      Action:
        required: true
        type: enum
        values: [CREATE, DELETE, MODIFY]
      Record.MY_FIELD:
        required: true
        maxLength: 64

    rules:
      # require same-sheet column when condition met
      - require: ActionKey
        when:
          column: Action
          operator: equals
          value: MODIFY
      # require cross-sheet (SheetName.ColumnName)
      - require: Node_Details.Node_Name
