CIQ Processor — Developer Guide

Validates CIQ Excel workbooks · Produces validation reports · Generates MOP JSON output

Overview

The CIQ Processor reads a Nokia CIQ (Configuration Input Questionnaire) Excel workbook, validates every data row against YAML-defined rules, writes validation reports, and — when validation passes — segregates the data into per-scope JSON folders consumed by mop-generator-utility.

CIQ Excel (.xlsx)

The entire pipeline runs in a single pass with no intermediate disk writes until the reports and optional MOP JSON are ready.

CLI Usage

java -jar ciq-processor-1.0.0-cli.jar --mode <mode> [options]

Modes:
  ciq-validate  (default)  Validate a CIQ Excel file against rules
  ciq-generate             Generate a blank CIQ template from rules

ciq-validate required:
  --ciq           <file>   CIQ Excel workbook (.xlsx)
  --node-type     <type>   Node type, e.g. SBC or MRF
  --activity      <name>   Activity name, e.g. ANNOUNCEMENT_LOADING
  --rules         <file>   YAML validation-rules file
  --output        <dir>    Output directory for validation reports

ciq-validate optional:
  --format                  <csv>    JSON,HTML,MSEXCEL (default: all three)
  --json-output-dir         <dir>    Output directory for JSON files (PASSED only)
  --json-output-config-file <file>   JSON output config (*_json-output.yaml)
  --report-template-name    <file>   HTML report template filename (enables template HTML)
  --report-template-path    <dir>    Directory containing the HTML report template

ciq-generate required:
  --node-type   <type>   Node type
  --activity    <name>   Activity name
  --rules       <file>   YAML validation-rules file

ciq-generate optional:
  --output      <dir>    Output directory (default: current directory)

Exit code:  0 = PASSED/SUCCESS,  1 = FAILED/ERROR
Report file base name is configured in report_output.filename in the rules YAML (default: {nodeType}_{activity}_VALIDATION_REPORT).

When --report-template-name is provided the HTML report is rendered from that external template file. If --report-template-path is also given it is combined as <path>/<name>. Omit both to use the built-in HTML layout. A ready-to-use template is at src/main/resources/validation-report-template.html.

Example — validate with external HTML template

java -jar ciq-processor-1.0.0-cli.jar \
  --ciq                    MRF_ANNOUNCEMENT_LOADING_CIQ.xlsx              \
  --node-type              MRF                                             \
  --activity               ANNOUNCEMENT_LOADING                           \
  --rules                  MRF_ANNOUNCEMENT_LOADING_validation-rules.yaml \
  --output                 target/processor-output                         \
  --format                  JSON,HTML,MSEXCEL                               \
  --json-output-dir         target/mop-json                                 \
  --json-output-config-file MRF_ANNOUNCEMENT_LOADING_json-output.yaml      \
  --report-template-name    validation-report-template.html                 \
  --report-template-path    src/main/resources

JSON Output Config File

MOP JSON output is fully YAML-driven via a standalone *_json-output.yaml file that is completely independent of the validation-rules YAML. It is passed at runtime via two CLI parameters — both must be provided for JSON output to run:

--json-output-dir         <dir>    # where to write the JSON file(s)
--json-output-config-file <file>   # e.g. MRF_ANNOUNCEMENT_LOADING_json-output.yaml

JSON output is produced only when validation passes. There are no hardcoded modes — the output structure, segregation key, and file naming are all declared in the config file. JsonTemplateEvaluator processes the data: template recursively, resolving _each directives and relational lookups against the in-memory CiqDataStore. The same config file is read by mop-generator-utility and network-command-executor-utility.

File naming convention: {NODE_TYPE}_{ACTIVITY}_json-output.yaml

output_mode: single        # single | individual (see below)

segregate_by:              # required when output_mode: individual
  sheet:  Index
  column: CRGroup          # distinct values of this column → one folder each
  as:     $cr              # variable name used in data template

data:                      # free-form YAML template
  nodeType: MRF
  activity: ANNOUNCEMENT_LOADING
  nodes:
    _each: "DISTINCT Index.Node AS $node"
    node:    $node
    crGroup: Index.CRGroup
    email:   "USER_ID.EMAIL WHERE USER_ID.CRGroup = Index.CRGroup"
    niamID:  "Node_Details.'NIAM NAME' WHERE Node_Details.Node_Name = $node"
    tableData:
      _each: "ANNOUNCEMENT_FILES WHERE GROUP = Index.GROUP"
      INPUT_FILE:           INPUT_FILE
      MRF_DESTINATION_PATH: MRF_DESTINATION_PATH

_each directive

SyntaxMeaning
"DISTINCT Sheet.Col AS $var"One array element per distinct column value; $var holds the current value
SheetNameOne array element per row of the sheet
"Sheet WHERE Col = value"One array element per matching row

Scalar value expressions

SyntaxMeaning
$varVariable bound by an enclosing _each or segregate_by
Sheet.ColumnFirst non-blank value from that sheet+column in the current scope
"Sheet.Col WHERE Sheet.Filter = $var"First matching value from filtered rows (relational lookup)

output_mode: single

One JSON file for the entire workbook. The data: template is evaluated once over the full data store.

JSON output layout

mop-json/ └── MRF_ANNOUNCEMENT_LOADING.json

output_mode: individual

One folder per distinct value of the segregate_by column; one JSON file per folder. The data: template is evaluated once per segregation value, with the as: variable bound to the current value.

Example — segregate by CRGroup

segregate_by:
  sheet:  Index
  column: CRGroup
  as:     $cr

JSON output layout

mop-json/ ├── CR-001/ │ └── MRF_ANNOUNCEMENT_LOADING_CR-001.json └── CR-002/ └── MRF_ANNOUNCEMENT_LOADING_CR-002.json

Output Parameters

When validation passes, parameters are printed to stdout as KEY=VALUE and returned via ValidationReport.getParameters(). All parameters (beyond the always-present STATUS and ERRORS) are declared in the outputs: section of the rules YAML — there are no hardcoded parameters.

Aggregates

aggregateDescriptionExample output
distinctSorted, deduplicated non-blank values joined by separatorCR_LIST=CR1,CR2
countCount of non-blank values in column; total rows when column omittedCR_COUNT=5
sumNumeric sum; non-numeric cells silently skippedTOTAL_KB=2048
groupFor each distinct column value emit <KEY>_<value> = collected groupBy valuesNODES_CR1=MRF1,MRF2

Example outputs: YAML

outputs:
  # Distinct CR numbers → CR_LIST=CR1,CR2
  CR_LIST:
    sheet: Index
    column: CRGroup
    aggregate: distinct
    separator: ","

  # Total rows in Index → CR_COUNT=4
  CR_COUNT:
    sheet: Index
    aggregate: count

  # Nodes per CRGroup → NODES_CR1=MRF1,MRF2   NODES_CR2=MRF3
  NODES:
    sheet:     Index
    column:    CRGroup
    aggregate: group
    groupBy:   Node
    separator: ","

The name: field overrides the YAML key as the emitted parameter name:

node_output:
  name: UNIQUE_NODES    # emits UNIQUE_NODES=... instead of NODE_OUTPUT=...
  sheet: Index
  column: Node
  aggregate: distinct

Validation Reports

Three report formats are supported, all written to --output:

FormatFileContents
JSON <name>.json Machine-readable; full error list with sheet, row, column, message and status
HTML <name>.html Human-readable; colour-coded per-sheet error tables with summary counts
MSEXCEL <name>.xlsx Excel report; one sheet per data sheet with error rows highlighted

Base name is configured in report_output.filename in the rules YAML (default: {nodeType}_{activity}_VALIDATION_REPORT). {nodeType} and {activity} are substituted from the CLI arguments.

HTML Report Template

By default the HTML report uses a built-in layout. To use a custom template, pass --report-template-name (and optionally --report-template-path):

--report-template-name  validation-report-template.html \
--report-template-path  src/main/resources

The template uses {{placeholder}} substitution and {{#section}}...{{/section}} loops. Available placeholders:

PlaceholderValue
{{status}}, {{nodeType}}, {{activity}}Report metadata
{{totalErrors}}, {{sheetsCount}}, {{passedSheets}}, {{failedSheets}}, {{totalRows}}Summary counts
{{generatedAt}}Timestamp
{{#sheets}}...{{/sheets}}Loop over sheet results
{{sheetName}}, {{sheetStatus}}, {{rowsChecked}}, {{errorCount}}Per-sheet values
{{sheetHeaderClass}}, {{sheetBadgeClass}}CSS class strings (pass/fail)
{{#errors}}...{{/errors}}Loop over errors in a sheet
{{rowNumber}}, {{column}}, {{value}}, {{message}}Per-error values
{{#if_passed}}...{{/if_passed}}, {{#if_failed}}...{{/if_failed}}Conditional blocks
{{#globalErrors}}{{message}}{{/globalErrors}}Index-level errors

A ready-to-use template is provided at src/main/resources/validation-report-template.html.

CIQ Excel Workbook Structure

The exact sheet and column structure is defined by the YAML rules file. The processor does not enforce a fixed set of sheet or column names. Typical structure:

SheetPurposeTypical columns
Index Maps nodes → groups/CRs Declared in sheets.Index in the YAML; any columns
Node_Details / NODE_ID NIAM (NE ID) mapping for every node Node name column + NIAM column (names configured in YAML)
<TableName> Configuration data (one sheet per table) Grouping column + record fields; declared in sheets.<TableName>
User_ID CR → responsible user email (optional) CRGroup + EMAIL

Action column values

ValueMeaningGenerated XML operation

DELETEDelete an existing recordxc:operation="delete"
MODIFYModify an existing recordxc:operation="merge"

CIQ Columns as MOP Template Variables

After segregation, the per-group JSON files are consumed by mop-generator-utility. Any data sheet column becomes a ${COLUMN_NAME} variable that can be referenced in the MOP template YAML commands. Two resolution strategies apply:

Column characteristicResolved asExample
Same value across all rows in the sheet (e.g. CIRCLE, VERSION) Group-level variable — substituted once, globally ${CIRCLE} → RJ
Different value per row (e.g. INPUT_FILE, MRF_DESTINATION_PATH) Per-row expansion — command is written once per CIQ row with the row's value ${INPUT_FILE} → one command per file in the CIQ table

Built-in variables always available regardless of CIQ data: ${NODE} (node name from the GroupIndex) and ${NEID} (NE identifier from the NIAM mapping).

Column names in the CIQ data sheet must match ${COLUMN_NAME} references in the MOP template exactly (case-sensitive). Use Record.-prefixed columns (Record.INPUT_FILE) without the prefix in the template reference (${INPUT_FILE}).

For MODIFY rows, ActionKey (which field to match on) and SubAction (ADD/DEL/MOD for sub-table entries) may be required depending on the rules.

report_output

Controls report file naming and output formats. {nodeType} and {activity} are substituted from the CLI arguments.

report_output:
  formats:  [JSON, HTML, MSEXCEL]
  filename: "{nodeType}_{activity}_VALIDATION_REPORT"
HTML template is NOT part of report_output — it is passed via CLI parameters --report-template-name and --report-template-path.

outputs

Post-validation aggregates computed only when validation PASSES. Each entry is added to ValidationReport.getParameters() and printed to stdout.

outputs:

  CR_LIST:
    sheet: Index
    column: CRGroup
    aggregate: distinct
    separator: ","

  # Row count → CR_COUNT=4

    sheet: Index
    aggregate: count

    sheet:     Index
    column:    CRGroup
    aggregate: group

Data Sheet Rules

Column reference convention — applies to all rule types:
ColumnName → column in the current sheet (default)
SheetName.ColumnName → column in the named sheet
This applies to require, forbid, when.column, compare, one_of, only_one, all_or_none, sum, and equals. 
sheets:

  MyTable:
    columns:

      Node:
        required: true
        crossRef:             # Value must exist in another sheet's column
          sheet: "_index"     # "_index" = the INDEX sheet
          column: node

      Action:
        required: true
        type: enum
        values: [CREATE, DELETE, MODIFY]

      ActionKey:
        requiredWhen:         # Required only when another column has a specific value
          column: Action
          value:  MODIFY

      SubAction:
        requiredWhen:
          column: Action
          value:  MODIFY
        type: enum
        values: [ADD, DEL, MOD]

      ID:
        required: true
        type: integer
        minValue: 1
        maxValue: 6000

      NAME:
        minLength: 1
        maxLength: 19

      Record.VERSION:
        required: true
        pattern: "^\\d+\\.\\d+\\.\\d+\\.\\d+$"
        messages:
          pattern: "VERSION must be in format X.X.X.X (e.g. 17.0.3.7)"

      Record.SOME_RANGE_FIELD:
        type: integer
        allowedRanges:        # Multiple allowed numeric bands (OR logic)
          - { min: 0,    max: 0    }
          - { min: 500,  max: 500  }
          - { min: 1024, max: 65535 }

      Tables:
        required: true
        sheetRef: true              # value must match an existing sheet name
        sheetRefIgnoreCase: false   # exact-case match (set true for case-insensitive)

    rules:

      # require a column when a same-sheet condition is met
      - require: ActionKey
        when:
          column: Action
          operator: equals          # equals | notEquals | blank | notBlank | contains
          value: MODIFY             # (not used for blank/notBlank)

      # forbid a column based on a condition
      - forbid: SubAction
        when:
          column: Action
          operator: equals
          value: CREATE

      # cross-sheet require — value must exist in referenced sheet's column
      - require: Node_Details.Node_Name

      # cross-sheet when — apply rule only when row's value exists in another sheet
      - require: NIAM_NAME
        when:
          sheet: Node_Details       # sheet to look up in
          column: Node_Name         # column whose value is looked up
          operator: exists          # exists | notExists

      # numeric/lexicographic when condition
      - require: SubAction
        when:
          column: Version
          operator: greaterThan     # greaterThan | greaterThanOrEquals | lessThan | lessThanOrEquals
          value: "17.0"

      # compound when condition (AND / OR nesting)
      - require: SubAction
        when:
          all:
            - column: Action
              operator: equals
              value: MODIFY
            - any:
                - column: SubAction
                  operator: blank
                - column: ActionKey
                  operator: notBlank

      # cross-column comparison — string operators only
      - compare: "StartPort lessThanOrEquals EndPort"

      # group presence
      - one_of: [IPv4_Address, IPv6_Address]
      - all_or_none: [SubAction, ActionKey]

      # column sum
      - sum: [Leg1, Leg2]
        equals: Total

Programmatic API

ValidationReport report = new CiqProcessorImpl().process(
    "/path/to/CIQ.xlsx",                           // ciqFilePath
    "MRF",                                         // nodeType
    "ANNOUNCEMENT_LOADING",                        // activity
    "/path/to/MRF_ANNOUNCEMENT_LOADING_validation-rules.yaml", // rulesFilePath
    "/path/to/output",                             // outputDir
    "JSON,HTML,MSEXCEL",                           // formatCsv  (null = all three)
    "/path/to/mop-json",                           // mopJsonOutputDir (null = skip)
    "validation-report-template.html",             // reportTemplateName (null = built-in)
    "src/main/resources"                           // reportTemplatePath (null = name is full path)
);

System.out.println(report.getStatus());        // "PASSED" or "FAILED"
System.out.println(report.getTotalErrors());   // total error count

// Output parameters declared in outputs: YAML section (populated on PASSED)
report.getParameters().forEach((k, v) -> System.out.println(k + "=" + v));

// Per-sheet errors
for (SheetValidationResult r : report.getSheets()) {
    System.out.println(r.getSheetName() + ": " + r.getErrors().size() + " error(s)");
}

Adding a New Node Type / Activity

No Java code changes are needed. Only a YAML file is required.
  1. Create {NODE_TYPE}_{ACTIVITY}_validation-rules.yaml.
  2. Configure the settings: block — set validateIndexSheets and validateNodeIds as needed.
  3. Define indexSheet.columns and nodeIdSheet.columns for the workbook layout.
  4. Define sheets: with column-level rules for each data sheet.
  5. Add a report_output: block to control the report file name and formats.
  6. Add an outputs: block to declare any console/post-validation aggregates (e.g. group, set, map).
  7. Create {NODE_TYPE}_{ACTIVITY}_json-output.yaml to describe how validated data is written to MOP JSON — set output_mode, segregate_by, and define the data template with _each directives. Pass it via --json-output-config-file.
  8. Pass the rules YAML path via --rules on the CLI, or as rulesFilePath in the API.

Validation rules YAML skeleton

# {NODE_TYPE}_{ACTIVITY}_validation-rules.yaml
